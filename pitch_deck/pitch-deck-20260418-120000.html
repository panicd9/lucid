THE SOLUTION

Read what you sign — on the device itself.

📜

Intent Definitions

Anchor IDL → human-readable ruleset. Every action templated.

🔒

Plain-English Signing

Signers approve via signMessage. The wallet renders the action as text — on the device, outside any compromised host.

⚡

On-Chain Verify

Program reconstructs the message from state and verifies via ed25519 precompile. Tamper → signature fails → reject.

Aligned with sRFC 38: Offchain Message Specification Version 1 — Anza, December 2025.

Speaker Notes
"Lucid uses signMessage instead of signTransaction. Any wallet that displays signed messages — Phantom, Backpack, hardware wallets — renders the action in plain English. The security guarantee gets stronger when signers use hardware wallets, because the host can't lie about what's on the device's own screen. The on-chain program reconstructs the expected message from state and verifies the ed25519 signature; if anything was tampered, the signature won't match. We're aligned with sRFC 38: Offchain Message Specification Version 1 — Anza's standard for human-readable offchain signing, published December 2025."

Speaker Notes
"The natural question — why doesn't Squads just add this? Because the verification doesn't live in the UI. Existing multisigs sign raw transaction bytes and trust the frontend to render a human-readable preview. Lucid's signers sign a structured message, and the program reconstructs the same bytes from on-chain state and validates the ed25519 signature against the reconstruction. Tamper anything between proposal and signature, and the signature won't verify. To add this, an existing multisig has to switch from signTransaction to signMessage and rebuild the approval flow on top of on-chain verification — that's a rewrite, not a feature."
